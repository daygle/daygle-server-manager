How to Use Daygle Server Manager

Getting Started

New to Daygle Server Manager? Follow these steps after the setup wizard completes.

  1. Go to SSH Keys and generate or import an SSH key pair. Ed25519 is recommended for new setups.
  2. Copy the public key to ~/.ssh/authorized_keys on each target server. Use the Copy authorized_keys entry button to get the correctly formatted line.
  3. Add your servers under Servers. Fill in the hostname, SSH port, username, and select your key. Use Test Connection to verify access before saving.
  4. Run a manual update from System Updates on a non-critical server first to confirm everything works end-to-end.
  5. Once satisfied, create recurring schedules under Scheduled Updates to automate patching on a cron timetable.
  6. Configure SMTP in Global Settings if you want email notifications for failures and reboot requirements.

See each section below for detailed guidance on every feature.

Dashboard

The dashboard shows a quick operational snapshot of your infrastructure:

  • Total configured servers - count of all registered hosts
  • Running and failed update jobs - current execution status and recent failures
  • Latest update activity - most recent jobs with timestamps

Status indicators use color coding: green = success, red = failed, amber/yellow = warning or in-progress, gray = pending or unknown.

The dashboard updates in real-time. Click any section or widget to jump to the detailed view for deeper analysis. Use Server Status page for comprehensive health checks across all servers.

Servers

Servers are your remote hosts that will receive package updates via SSH. Go to Servers to add or manage them.

  • Name: a unique, human-readable display name (e.g., "prod-web-01", "staging-db")
  • Host: IP address (192.168.1.50) or DNS hostname (prod-server.example.com)
  • Port: SSH port (default 22; use a different port if your server uses a custom SSH port)
  • Username: the SSH account that will run updates (must have sufficient permissions)
  • Auth Method: choose SSH Key (recommended for security) or Password
  • Sudo Password: only needed if your SSH user requires a different password to run sudo commands

Best practice: Always use SSH keys instead of passwords for security. Test the connection before saving using the Test Connection button.

Use Edit and Delete buttons in the table to modify or remove servers. Reboot status indicator (amber icon) shows servers that require a reboot after updates.

System Updates (Run Now)

Run immediate package updates on one or more servers. Updates execute immediately after you submit the form.

  • Select servers: check one or multiple targets. Useful for staged rollouts (test non-critical servers first, then production)
  • Choose mode: "Check for Updates" scans for available packages without installing (alert-only), or "Start Update" installs packages
  • Package manager: select Auto detect to auto-identify (apt/dnf/yum), or manually choose if you know the system
  • APT extra steps: (if available) optionally enable additional cleanup steps like autoremove and clean

Monitoring updates: the Update Jobs table auto-refreshes every few seconds. Click any job row to expand full output and follow progress in real-time. Updates typically take 5–30+ minutes depending on package count and server load.

For recurring updates, use Update Schedules instead.

Update Schedules

Automate recurring updates on a cron-based timetable. Go to System Updates → Schedules to create or manage them.

  • Schedule name: a descriptive name (e.g., "prod-nightly-2am", "test-weekly")
  • Cron expression: a 5-field format controlling execution time
  • Target servers: select which servers run this schedule
  • Package manager: auto-detect or manually specify

Cron Field Reference:

minute hour day month day-of-week where each field is 0–59, 0–23, 1–31, 1–12, 0–6 (0/7=Sunday)

Examples:

0 2 * * *      # daily at 02:00 (2 AM)
0 */6 * * *    # every 6 hours (at 00:00, 06:00, 12:00, 18:00)
30 3 * * 0     # every Sunday at 03:30 (3:30 AM)
0 0 1 * *      # first day of each month at midnight
15 22 * * 1-5  # weekdays at 22:15 (10:15 PM)

Timezone: schedules run in the timezone you were in when creating the schedule (based on your Global Settings at that time). Override system-wide in Global Settings.

Management: use Enable/Disable to pause without deleting, or Delete to remove. Each schedule runs independently.

Alerts

Alerts are automatically created when update jobs fail or when reboot-required is detected. Manage them in the Alerts page.

  • Severity levels: Info (informational), Warning (needs attention), Error (action required)
  • States: Open (unhandled), Acknowledged (someone is working on it), Dismissed (resolved or false alarm)
  • Filter: by severity, state, date range, or search by server name or message
  • Acknowledge: mark an alert to let your team know you're handling it
  • Dismiss: permanently remove an alert (use for resolved issues)

Notifications: admin users see an unread alert count badge in the sidebar. If email alerts are enabled, admins receive email notifications for new alerts.

Best practice: regularly acknowledge or dismiss alerts to keep the queue clean and prevent alert fatigue.

Audit Log

Audit Log is a security and compliance record of all important system changes, user actions, and events.

What's logged:

  • User logins and logouts
  • Server configuration changes (add, edit, delete)
  • Job execution and failures
  • Alert management (acknowledge, dismiss)
  • SSH key usage and changes
  • User and settings modifications
  • Reboot commands and system actions

Search and Filter:

  • Search by text: find any server name, username, action, or detail across all entries
  • Filter by action: see only specific types of events (e.g., job.run, server.add, user.login)
  • Filter by actor: show actions by a specific user
  • Filter by date range: narrow results to a specific timeframe

Retention: audit entries are kept according to your History Retention setting in Global Settings. Older entries are automatically deleted.

Use cases: find who deleted a server, when an SSH key was last used, track failed login attempts, or generate compliance reports.

User Management Admin

Only admin users can access User Management. Create, edit, enable/disable, or delete users here.

User Roles:

  • Admin: can access settings, create/delete users, manage SSH keys, and view all alerts
  • Non-admin: can view servers and job history but cannot modify configuration

Actions:

  • Create user: enter username, password, email address, and admin status
  • Edit: change profile info, admin status, or enabled/disabled state
  • Enabled toggle: quickly suspend a user without deleting them
  • Email address: required for users who should receive alert emails
  • Delete: permanently remove a user account

Safety Rules:

  • Cannot delete the last enabled admin user
  • Cannot disable/delete your own admin account while logged in
  • Always keep at least one admin user enabled for system management

SSH Keys Admin

SSH keys are more secure than passwords and recommended for production servers. Store and manage them here.

Key Management:

  • Generate new key: choose key type (RSA 4096 or Ed25519 - both secure) to generate a keypair in-system
  • Upload existing key: paste or upload a private key you already have on your servers
  • View fingerprint: use the displayed fingerprint to verify the key matches what's on the server
  • Store securely: private keys are encrypted and stored safely in the database

Best Practices:

  • Use different keys for different servers or environments (dev vs. prod)
  • Never share or download private keys from this system - generate them once and deploy to servers
  • Rotate keys periodically: generate new key, test on a server, update server reference, delete old key
  • Use Ed25519 keys for new setups (smaller, faster, equally secure)

Deletion: you cannot delete a key if it's assigned to any server. Unassign all servers using that key first.

Global Settings Admin

Global Settings control system-wide defaults and behavior. Individual users can override most settings in their User Settings.

  • Date/time display format: affects how dates and times are shown throughout the system (ISO, US 12-hour, European, month-name, etc.)
  • Default timezone: used for schedule execution, alert timestamps, and new user profiles. Users can override individually.
  • History retention: how long to keep job records, audit logs, and alerts before automatic deletion. Balance storage with compliance needs.
  • Login timeout: minutes of inactivity before users are automatically logged out (e.g., 30 minutes for security)
  • Email alerts: enable/disable all alert email delivery system-wide
  • SMTP configuration: mail server hostname, port (usually 587 or 465), TLS enabled/disabled, username, and password
  • Sender address & name: appears in the From field of alert emails (e.g., "alerts@example.com" or "Daygle Alerts")
  • SMTP test: verify connectivity to your mail server before saving (recommended before enabling alerts)

User Settings

Personalize your experience by setting your own date/time format and timezone. Leave blank to use Global Settings defaults.

  • Date format: choose how you prefer dates displayed (MM/DD/YYYY, DD/MM/YYYY, ISO, month names, etc.)
  • Timezone: your local timezone affects job output timestamps and schedule display in your view

Use case: "Our team is distributed: I'm in PT but our servers and schedules use UTC - I set my timezone to PT to see times in my local offset."

Email Alert Delivery

When email alerts are enabled in Global Settings, alert messages are sent to admin users who have an email address configured in User Management.

Alert Triggers:

  • Job failures (update failed, reboot command failed, etc.)
  • Reboot-required detection after successful updates
  • Critical system events or warnings

Email Content:

  • Server name and issue description
  • Timestamp (in your personal timezone if set)
  • Direct link to the alert in the system for quick action

Configuration:

  • Ensure SMTP settings are correct in Global Settings
  • Run the SMTP test before enabling alerts
  • Each admin user needs an email address in their User Management profile
  • Tip: use a distribution list (ops-team@example.com) for team notifications

Alert fatigue: regularly acknowledge or dismiss handled alerts to keep the system responsive and notifications meaningful.

Common Failure Checks

Authentication failed

  • Verify SSH username is correct
  • Test manually: ssh -i /path/to/key user@host or ssh user@host
  • If using password auth, ensure the password is correct
  • For key auth, ensure the private key matches the public key on the server

Sudo failed

  • Verify user has sudo privileges: sudo -l on the server
  • If sudo requires a password, fill in the Sudo Password field
  • Check if the user is in the sudo group: groups user
  • Run visudo to verify sudoers configuration is correct

Package manager unknown

  • Server may not have apt/dnf/yum installed
  • Test manually: which apt, which yum, or which dnf
  • Try with Auto Detect first on a single non-critical server to identify the system
  • If unsure, manually select the package manager after identifying the Linux distribution

Timeout / Unreachable

  • Check connectivity: ping host or test DNS: nslookup hostname
  • Verify SSH port is open: nc -zv host 22
  • Check firewall rules (allow SSH port inbound from your system)
  • Verify VPN connection if accessing remote networks
  • Ensure the server is powered on and not in a maintenance window

SMTP test failed / Email alerts not sending

  • Verify SMTP hostname and port (usually 587 for TLS, 465 for SSL)
  • Ensure TLS/SSL setting matches your mail server config
  • Test credentials manually if possible
  • Check firewall allows outbound email (port 587 or 465)
  • Verify sender address is valid and accepted by your mail server
  • Run SMTP test in Global Settings before enabling alerts

General debugging

  • Check browser console (F12 Developer Tools) for client-side errors
  • Review job output in Update Jobs table for detailed error messages
  • Check server-side logs if you have SSH access to the application server
  • Review Audit Log to see what actions triggered the failure

Update This Application

Keep Daygle Server Manager up to date with the latest features and security fixes.

⚠︎ Before updating: backup your database and configuration files.

bash update.sh --check
# Check available updates without installing

bash update.sh
# Install available updates (recommended)

bash update.sh --force
# Reinstall even if already up to date (use if installation incomplete)

bash update.sh --skip-start
# Update without restarting services (useful for coordination)

bash update.sh --help
# Show all available options

Update Process:

  • Duration: typically 1–5 minutes depending on what's being updated
  • Downtime: minimal; services are restarted automatically unless --skip-start is used
  • After update: verify server status on the dashboard, run a test update on a non-critical server first if possible
